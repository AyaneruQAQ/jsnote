# 项目技术栈分析与面试准备文档

## 1. 技术栈分析

### 1.1 前端框架及版本
- **Vue 3.5.13**：采用最新的Vue 3 Composition API，提供更好的类型支持和组件复用能力
- **TypeScript 5.6.2**：全面使用TypeScript进行类型安全开发，提高代码质量和可维护性

### 1.2 构建工具及配置
- **Vite 5.1.4**：现代化的前端构建工具，提供快速的开发体验和高效的构建输出
- **配置亮点**：
  - 自定义路径别名 `@` 指向 `src` 目录，便于模块导入
  - 丰富的插件生态：UnoCSS、AutoImport、Components、SVG Icons等
  - 多环境代理配置，支持开发、测试和生产环境
  - 优化的依赖预构建，排除了常用Element Plus组件的样式依赖

### 1.3 状态管理方案
- **Pinia 2.1.7**：Vue 3官方推荐的状态管理库，提供更简洁的API和更好的TypeScript支持
- **使用场景**：
  - 全局用户状态管理
  - 待办任务数据管理
  - 工作台核心数据共享
  - 配置信息管理

### 1.4 UI组件库选择及使用情况
- **Element Plus 2.4.2**：作为主要UI组件库，提供丰富的基础组件
- **Vxe Table 4.7.77** + **Vxe PC UI 4.0.62**：专业的表格解决方案，处理复杂的表格场景
- **UnoCSS**：原子化CSS框架，提高样式开发效率和一致性
- **组件自动注册**：通过 `unplugin-vue-components` 实现组件的自动导入和注册

### 1.5 路由管理方案
- **Vue Router 4**：官方路由管理器，支持Composition API
- **路由结构**：
  - 工作台首页
  - 待办任务列表与详情
  - 事件列表与详情
  - 流程管理
  - 表单模板管理
  - 报表页面
- **路由守卫**：实现页面标题动态设置和白名单跳转逻辑

### 1.6 网络请求处理方式
- **Axios**：基于Promise的HTTP客户端，用于处理API请求
- **请求封装**：
  ```typescript
  export const request = (params: AxiosRequestConfig, options: Options = {}) => {
    // 统一请求处理逻辑
  };
  ```
- **响应拦截**：处理401登录失效重定向和统一错误提示
- **类型定义**：自定义 `WBResponse` 类型，统一API响应格式

### 1.7 代码规范与质量控制工具
- **ESLint 9.15.0**：使用公司内部定制的ESLint配置 `@boss/eslint-config`
- **Prettier 3.3.3**：代码格式化工具，保持代码风格一致性
- **Vue-tsc 2.1.10**：Vue 3 TypeScript编译器，进行类型检查
- **自动修复脚本**：提供 `lint:fix` 和 `format` 命令，便于快速修复代码问题

### 1.8 测试框架及覆盖率情况
- **Vitest 1.6.1**：现代化的前端测试框架，与Vite深度集成
- **测试工具**：
  - `@vue/test-utils`：Vue组件测试工具
  - `@testing-library/vue`：基于DOM的测试库
  - `happy-dom` + `jsdom`：模拟浏览器环境
- **测试脚本**：
  - `test`：启动Vitest测试
  - `test:ui`：启动可视化测试界面
  - `test:run`：运行所有测试
  - `coverage`：生成测试覆盖率报告

### 1.9 其他关键技术或库
- **UnoCSS**：原子化CSS框架，提高样式开发效率
- **SVG Icons**：通过 `vite-plugin-svg-icons` 统一管理SVG图标
- **Auto Import**：通过 `unplugin-auto-import` 自动导入常用API
- **Sass**：CSS预处理器，支持嵌套、变量等特性
- **Day.js**：轻量级日期处理库
- **Lodash**：实用工具库，提供常用的工具函数
- **Nanoid**：生成唯一ID的库
- **X6**：图编辑引擎，用于流程可视化

## 2. 面试问答部分

### 2.1 技术选型相关问题

#### Q1：为什么选择Vue 3 + TypeScript作为技术栈？
**核心要点**：
- 技术先进性：Vue 3提供了更好的性能和TypeScript支持
- 开发效率：Composition API便于逻辑复用和代码组织
- 类型安全：TypeScript减少运行时错误，提高代码可维护性
- 生态成熟：Vue 3生态已经成熟，有丰富的插件和工具支持
- 团队熟悉度：团队成员对Vue技术栈较为熟悉

**展开说明**：
```typescript
// Vue 3 Composition API示例
import { ref, computed, onMounted } from 'vue';

export default {
  setup() {
    const count = ref(0);
    const doubleCount = computed(() => count.value * 2);
    
    onMounted(() => {
      console.log('Component mounted');
    });
    
    return {
      count,
      doubleCount
    };
  }
};
```

#### Q2：为什么选择Vite作为构建工具？
**核心要点**：
- 快速的开发体验：基于ES Module的热更新，启动速度快
- 高效的构建输出：使用Rollup进行生产构建，输出体积小
- 现代的插件系统：支持多种插件，便于扩展功能
- 良好的TypeScript支持：内置TypeScript编译，无需额外配置
- 开箱即用的配置：默认配置已经满足大部分需求，减少配置复杂度

**展开说明**：
Vite的热更新机制基于原生ES Module，避免了Webpack的全量编译，开发时启动速度和热更新速度都有明显提升。生产构建使用Rollup，通过Tree Shaking和代码分割等技术，输出更小的bundle体积。

#### Q3：为什么选择Pinia而不是Vuex？
**核心要点**：
- 更简洁的API：不需要mutations，直接修改state
- 更好的TypeScript支持：自动推断类型，无需手动定义
- 模块化设计：每个store都是独立的模块，便于维护
- 支持Composition API：可以在store中使用Composition API
- 更小的体积：比Vuex更轻量

**展开说明**：
```typescript
// Pinia Store示例
import { defineStore } from 'pinia';

export const useCounterStore = defineStore('counter', {
  state: () => ({
    count: 0
  }),
  actions: {
    increment() {
      this.count++;
    }
  },
  getters: {
    doubleCount: (state) => state.count * 2
  }
});
```

### 2.2 项目架构设计相关问题

#### Q1：项目的组件化设计思路是什么？
**核心要点**：
- 分层设计：基础组件、业务组件、页面组件分离
- 高内聚低耦合：组件职责单一，便于复用和维护
- Props/Events通信：组件间通过Props传递数据，Events触发事件
- 组合式API：使用Composition API提高组件逻辑的复用性
- 类型安全：全面使用TypeScript定义组件的Props和Events

**展开说明**：
项目组件分为三层：
1. **基础组件**：通用的UI组件，如按钮、表单、表格等
2. **业务组件**：与业务相关的可复用组件，如待办任务卡片、流程步骤组件等
3. **页面组件**：页面级别的组件，负责组织和协调业务组件

#### Q2：项目的状态管理设计是怎样的？
**核心要点**：
- 模块化store设计：按功能划分store，如用户store、待办store等
- 状态分层：全局状态、页面状态、组件状态分离
- 异步处理：在store的actions中处理异步请求
- 响应式更新：利用Vue 3的响应式系统，实现状态变化自动更新UI
- 状态持久化：部分关键状态通过localStorage持久化

**展开说明**：
项目使用Pinia进行状态管理，按功能划分为多个store：
- `user.ts`：用户信息管理
- `workbench.ts`：工作台核心数据
- `soToDo.ts`：待办任务数据
- `taskData.ts`：任务详情数据
- `config.ts`：配置信息管理

#### Q3：项目的路由设计有什么特点？
**核心要点**：
- 清晰的路由结构：按功能模块划分路由
- 动态路由：支持参数化路由，如待办详情页
- 路由守卫：实现页面标题设置、权限控制等
- 懒加载：路由组件按需加载，提高首屏加载速度
- 元信息：通过meta字段配置路由的额外信息

**展开说明**：

### 2.4 性能优化相关问题

#### Q1：项目中采取了哪些性能优化措施？
**核心要点**：
- 按需加载：路由组件和第三方库按需加载
- 代码分割：将代码分割为多个chunk，减少初始加载体积
- 图片优化：使用合适的图片格式和尺寸，支持懒加载
- 虚拟滚动：处理大数据量列表和表格
- 缓存策略：缓存API响应和组件状态
- 减少重绘重排：优化DOM操作，使用CSS过渡动画

**展开说明**：
项目使用Vite的代码分割功能，将代码分割为多个chunk，实现按需加载。对于大数据量的列表和表格，使用虚拟滚动技术，只渲染可视区域内的内容。图片资源使用合适的格式和尺寸，并支持懒加载。

#### Q2：如何优化首屏加载速度？
**核心要点**：
- 减少初始加载体积：按需加载第三方库，代码分割
- 预加载关键资源：使用link preload预加载关键资源
- 优化资源加载顺序：CSS放在head，JS放在body底部
- 使用CDN：将静态资源部署到CDN，提高加载速度
- 缓存策略：合理设置缓存头，缓存静态资源

**展开说明**：
项目在构建时使用Vite的优化功能，将第三方库分割为独立的chunk，实现按需加载。同时，使用CDN部署静态资源，提高资源加载速度。通过合理的缓存策略，减少重复请求，提高页面加载速度。

#### Q3：如何优化组件的渲染性能？
**核心要点**：
- 使用v-once：对于静态内容使用v-once指令
- 合理使用v-memo：缓存组件的渲染结果
- 避免不必要的计算属性：只在依赖变化时才重新计算
- 虚拟滚动：处理大数据量列表
- 组件拆分：将大型组件拆分为多个小型组件

**展开说明**：
项目中使用了多种Vue优化手段，如v-once、v-memo等指令，减少不必要的组件渲染。对于大数据量的列表和表格，使用虚拟滚动技术，只渲染可视区域内的内容。同时，将大型组件拆分为多个小型组件，提高组件的渲染性能。
